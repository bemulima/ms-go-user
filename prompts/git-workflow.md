# Правила работы с Git и GitHub Issues

## Процесс работы с задачами

### 1. Получение задачи

После получения задачи необходимо:

1. ✅ Проверить, указан ли `#id` issue в обсуждении задачи
2. ✅ Если `#id` issue не указан, проверить, создано ли issue для этой задачи
3. ✅ Если issue не создано, создать issue на GitHub
4. ✅ Установить подходящий label (например: `bug`, `feature`, `refactoring`, `documentation`)
5. ✅ Установить себя как исполнителя (assignee)
6. ✅ Добавить процесс работы в виде checklist с шагами, которые нужно выполнить

Обязательное правило:
- Каждая задача выполняется только через отдельное issue и отдельную ветку.
- Прежде чем писать код или менять файлы, нужно проверить, есть ли `#id` issue в обсуждении задачи.
- Если `#id` issue не был указан в обсуждении, нужно проверить, создано ли issue для этой задачи.
- Если issue для задачи не создано, нужно создать issue на GitHub.
- Созданное или найденное issue нужно назначить issue этой задачи.
- Все коммиты по задаче должны быть связаны с этим issue через `(#id)` или `(#1,#2,#3)` в конце первой строки коммита.
- Запрещено выполнять задачи прямо в `main`/`dev`.
- Все изменения по задаче должны быть сделаны внутри ветки задачи, связанной с issue.
- По умолчанию ветка задачи всегда создаётся от актуальной ветки `main`.
- Ветку задачи можно создавать от `dev` только по явной команде пользователя, когда сказано создать ветку от `dev`.
- Pull Request создаётся обязательно для каждой задачи.
- В Pull Request нужно назначить себя исполнителем (assignee).
- Pull Request должен унаследовать все labels из связанного issue.
- Pull Request должен получить тот же milestone, что и связанное issue.
- В описании Pull Request обязательно указать, что было выполнено.
- Перед началом работы нужно проверить текущую git-ветку и наличие Pull Request для неё.
- Если текущая ветка не `main` и Pull Request для неё ещё не создан, нужно предупредить пользователя:
  - текущая ветка не `main`;
  - ветка ещё не отправлена в Pull Request.
- После такого предупреждения нельзя продолжать работу автоматически.
- Нужно дождаться решения пользователя:
  - остаться в текущей ветке и продолжить работу;
  - или остановиться и ждать отдельной команды, что делать дальше.
- Если текущая задача ещё не завершена и приходит следующая задача, нужно предупредить пользователя:
  - текущая задача ещё не завершена;
  - нужно решить, что делать с текущей задачей.
- После такого предупреждения нельзя автоматически начинать новую задачу.
- Нужно дождаться решения пользователя:
  - завершить текущую задачу;
  - приостановить текущую задачу и начать новую;
  - отменить текущую задачу;
  - или остановиться и ждать отдельной команды.
- После обсуждения задачи нельзя автоматически начинать писать код или менять файлы.
- Реализацию можно начинать только после явной команды пользователя, например:
  - `начинай`;
  - `можешь начать`;
  - `можешь реализовать`;
  - `приступай к реализации`.
- До такой команды разрешено только обсуждать, анализировать и уточнять план.

### 2. Структура issue

#### Для обычных задач:

```markdown
## Описание задачи
[Описание того, что нужно сделать]

## Процесс работы
- [ ] Шаг 1: Описание первого шага
- [ ] Шаг 2: Описание второго шага
- [ ] Шаг 3: Описание третьего шага
- [ ] Тестирование
- [ ] Code review
```

#### Для больших задач:

Если задача большая и требует разбиения на подзадачи:

1. ✅ Создать основное issue с общим описанием
2. ✅ Создать подзадачи (subtasks) под основным issue:
   - Каждая подзадача должна иметь свой label
   - Установить себя как исполнителя для каждой подзадачи
   - Добавить checklist для каждой подзадачи
3. ✅ Связать подзадачи с основным issue

Пример структуры:
```
#4 Большая задача: Рефакторинг архитектуры
  ├── #5 Подзадача 1: Создать структуру модулей
  ├── #6 Подзадача 2: Рефакторинг модуля X
  └── #7 Подзадача 3: Обновить документацию
```

### 3. Комментирование прогресса

**Важно:** Каждый раз, когда выполнена какая-то часть процесса, необходимо:

- ✅ Отметить выполненный шаг в checklist
- ✅ Добавить комментарий в issue с описанием выполненной работы
- ✅ Указать, что было сделано и какие изменения внесены

Пример комментария:
```markdown
✅ Выполнен шаг 1: Создана структура директорий

Созданы следующие директории:
- `src/modules/`
- `src/shared/ui-kit/`
- `src/app-runtime/`

Все директории следуют архитектуре из документации.
```

### 4. Формат коммитов (Commit Message Convention)

Мы следуем [Conventional Commits](https://www.conventionalcommits.org/) спецификации для наших commit сообщений.

#### Формат:

```
<type>: <subject> (#issue-id)

[optional body]

[optional footer]
```

Если коммит относится к нескольким issue/task, в конце первой строки указываются все номера через запятую без пробелов:

```
<type>: <subject> (#1,#2,#3)
```

#### Типы коммитов:

- **fix:** исправления багов, например: `fix: correct dark theme button color (#4)`
- **feat:** новые функции, например: `feat: add snackbar component (#5)`
- **refactor:** рефакторинг кода, например: `refactor: reorganize component folders (#6)`
- **docs:** изменения в документации, например: `docs: add button usage example (#7)`
- **test:** добавление или обновление тестов, например: `test: add button unit tests (#8)`
- **chore:** изменения в инструментах/конфигурации, например: `chore: update circleci config (#9)`

#### Обязательные требования:

1. ✅ **Всегда указывать номер issue/task** в формате `(#id)` в скобках в конце первой строки коммита (например: `(#4)`)
2. ✅ Если коммит относится к нескольким issue/task, указывать все номера через запятую без пробелов: `(#1,#2,#3)`
3. ✅ Использовать императив в сообщении (например: "add" вместо "added" или "adding")
4. ✅ Первая строка должна быть не длиннее 72 символов
5. ✅ Отделять тело коммита от заголовка пустой строкой

#### Примеры правильных коммитов:

```bash
# Простой коммит
feat: add user authentication module (#4)

# Коммит для нескольких issue/task
fix: align auth redirects with middleware rules (#4,#7,#9)

# Коммит с телом
refactor: reorganize project structure (#4)

- Created modules/ directory for domain modules
- Moved UI components to shared/ui-kit/
- Added app-runtime for orchestration

# Коммит с BREAKING CHANGE
feat: change button API (#4)

BREAKING CHANGE: Button component now requires onClick prop
```

#### Примеры неправильных коммитов:

```bash
# ❌ Нет типа
Added new feature (#4)

# ❌ Нет номера issue
feat: add authentication

# ❌ Несколько issue указаны с пробелами
fix: align auth redirects (#4, #7)

# ❌ Несколько issue указаны не в конце первой строки
fix: align auth redirects
(#4,#7)

# ❌ Неправильный формат issue (без скобок)
feat: add authentication #4

# ❌ Неправильный формат issue (без скобок в конце)
feat: add authentication
#4

# ❌ Неправильный формат issue (другие скобки)
feat: add authentication [4]

# ❌ Слишком длинное сообщение
feat: add a very long description that exceeds 72 characters limit and makes it hard to read (#4)
```

### 5. Работа с ветками и Pull Request

- Перед началом работы проверить текущую ветку
- Перед написанием кода или изменением файлов проверить, указан ли `#id` issue в обсуждении задачи
- Если `#id` issue не указан, проверить наличие issue для этой задачи; если issue нет, создать его
- Использовать найденное или созданное issue как обязательный issue id для всех коммитов задачи
- Если текущая ветка не `main`, проверить, есть ли Pull Request для этой ветки
- Если текущая ветка не `main` и Pull Request ещё не создан, обязательно предупредить пользователя и дождаться решения
- Если текущая задача ещё не завершена и приходит новая задача, обязательно предупредить пользователя и дождаться решения, что делать с текущей задачей
- После обсуждения не начинать реализацию автоматически: дождаться явной команды `начинай`, `можешь начать`, `можешь реализовать` или `приступай к реализации`
- По умолчанию всегда создавать ветку от актуальной ветки `main`
- Создавать ветку от `dev` только по явной команде пользователя
- Называть ветку по типу задачи: `fix/issue-4`, `feat/issue-5`, `refactor/issue-6`
- Всегда создавать Pull Request с ссылкой на issue в описании
- Если рабочая ветка создана от `main`, Pull Request отправлять в `main`
- Если рабочая ветка создана от `dev`, Pull Request отправлять в `dev`
- Если Pull Request отправлен в `dev`, обязательно предупредить пользователя, что PR отправлен на ветку `dev`
- В Pull Request обязательно назначить себя как assignee
- В Pull Request обязательно перенести все labels из связанного issue
- В Pull Request обязательно установить milestone из связанного issue
- В описании Pull Request обязательно указать, что было выполнено

### 6. Завершение задачи

После выполнения всех шагов:

1. ✅ Все шаги в checklist отмечены
2. ✅ Все изменения закоммичены с правильным форматом
3. ✅ Создан Pull Request
4. ✅ Добавлен финальный комментарий в issue с описанием выполненной работы
5. ✅ Issue закрыт (или готов к закрытию после merge)

### 7. Команды `комит пуш pr` и `комит пуш pr и прими в main`

Команды могут прийти в русской, английской или смешанной форме.
Нужно понимать их по смыслу:

- `комит пуш pr`
- `commit push pr`
- `commit && push`
- `commit && push && pr`
- `commit && push && create PR`
- `комит пуш pr и прими в main`
- `commit && push && pr && merge to main`

Если пользователь даёт команду `комит пуш pr` или английский эквивалент без merge в `main`, это означает:

1. ✅ Проверить изменения и сгруппировать их по логическим слоям/частям задачи.
2. ✅ Создать отдельные коммиты для каждой группы.
3. ✅ Каждый коммит оформить по правилам из раздела "Формат коммитов".
4. ✅ Запушить ветку в origin.
5. ✅ Создать Pull Request в правильную base-ветку:
   - если рабочая ветка создана от `main`, создать PR в `main`;
   - если рабочая ветка создана от `dev`, создать PR в `dev` и предупредить пользователя, что PR отправлен на ветку `dev`.

Группировка коммитов:

- тесты отдельно;
- domain отдельно;
- routes отдельно;
- infrastructure отдельно;
- UI отдельно;
- документация отдельно;
- конфигурация/инструменты отдельно.

Примеры группировки:

```bash
test: add auth flow coverage (#4)
refactor: update auth domain model (#4)
feat: add auth routes (#4)
feat: wire auth infrastructure (#4)
docs: update auth workflow notes (#4)
```

Если пользователь даёт команду `комит пуш pr и прими в main` или английский эквивалент с merge в `main`, это означает:

Если рабочая ветка создана от `main`:

1. ✅ Выполнить все шаги команды `комит пуш pr`.
2. ✅ После создания Pull Request принять (merge) этот Pull Request в `main`.
3. ✅ Перейти в ветку `main`.
4. ✅ Обновить локальную `main`.
5. ✅ Удалить рабочую ветку локально.
6. ✅ Удалить рабочую ветку в `origin`.
7. ✅ Добавить итоговый комментарий в связанное issue с описанием выполненной работы и ссылкой на PR.
8. ✅ Закрыть связанное issue.

Если рабочая ветка создана от `dev`:

1. ✅ Выполнить все шаги команды `комит пуш pr`, но Pull Request отправить в `dev`.
2. ✅ Предупредить пользователя, что первый PR отправлен на ветку `dev`.
3. ✅ Принять (merge) Pull Request рабочей ветки в `dev`.
4. ✅ Перейти в ветку `dev` и обновить локальную `dev`.
5. ✅ Создать Pull Request из `dev` в `main`.
6. ✅ В описании Pull Request `dev` -> `main` указать, что было выполнено.
7. ✅ Принять (merge) Pull Request `dev` -> `main`.
8. ✅ Перейти в ветку `main`.
9. ✅ Обновить локальную `main`.
10. ✅ Удалить рабочую ветку локально.
11. ✅ Удалить рабочую ветку в `origin`.
12. ✅ Добавить итоговый комментарий в связанное issue с описанием выполненной работы и ссылкой на PR.
13. ✅ Закрыть связанное issue.

Правила:

- Нельзя объединять несвязанные слои в один коммит, если их можно безопасно разделить.
- Нельзя создавать один общий коммит на все изменения, если изменения относятся к разным слоям.
- Все коммиты должны содержать issue/task id в конце первой строки: `(#id)` или `(#1,#2,#3)`.
- Pull Request после push обязателен.
- После merge Pull Request обязательно добавить итоговый комментарий в связанное issue и закрыть issue.

## Чеклист для новой задачи

- [ ] Создан issue на GitHub
- [ ] Проверено, указан ли `#id` issue в обсуждении задачи
- [ ] Если `#id` issue не указан, проверено наличие issue для этой задачи
- [ ] Если issue для задачи не было, создано новое issue
- [ ] Issue этой задачи выбран и используется во всех коммитах
- [ ] Установлен подходящий label
- [ ] Установлен исполнитель (assignee)
- [ ] Добавлен процесс работы (checklist)
- [ ] Если задача большая - созданы подзадачи
- [ ] Проверена текущая git-ветка
- [ ] Если текущая ветка не `main`, проверено наличие Pull Request для неё
- [ ] Если текущая ветка не `main` и Pull Request не создан, пользователь предупреждён и принято решение
- [ ] Если предыдущая задача не завершена, пользователь предупреждён и принято решение, что делать с текущей задачей
- [ ] Получена явная команда на старт реализации: `начинай`, `можешь начать`, `можешь реализовать` или `приступай к реализации`
- [ ] Создана ветка для работы от `main` по умолчанию или от `dev` только по явной команде
- [ ] Коммиты следуют Conventional Commits формату
- [ ] В каждом коммите указан issue/task id в конце первой строки: `(#id)` или `(#1,#2,#3)`
- [ ] Создан Pull Request
- [ ] Pull Request отправлен в правильную base-ветку: `main` для ветки от `main`, `dev` для ветки от `dev`
- [ ] Если Pull Request отправлен в `dev`, пользователь предупреждён, что PR отправлен на ветку `dev`
- [ ] В Pull Request назначен assignee
- [ ] В Pull Request перенесены labels из issue
- [ ] В Pull Request установлен milestone из issue
- [ ] В описании Pull Request указано, что было выполнено
- [ ] При команде `комит пуш pr` / `commit && push` изменения сгруппированы в отдельные коммиты по слоям
- [ ] При команде `комит пуш pr` / `commit && push` ветка запушена и Pull Request создан, если команда подразумевает PR
- [ ] При команде `комит пуш pr и прими в main` Pull Request принят в `main`
- [ ] Если рабочая ветка была от `dev` и пришла команда `комит пуш pr и прими в main`, рабочая ветка сначала принята в `dev`, затем создан и принят Pull Request `dev` -> `main`
- [ ] При команде `комит пуш pr и прими в main` выполнен переход в `main`, удалена рабочая ветка локально и в `origin`
- [ ] После merge Pull Request добавлен итоговый комментарий в issue
- [ ] После merge Pull Request связанное issue закрыто
- [ ] Прогресс комментируется в issue
- [ ] После завершения - issue закрыт
